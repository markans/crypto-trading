Bybit Signal Workflow
=====================

Purpose
-------

Use this workflow when you want Codex to analyze a Bybit symbol and prepare or
place a Bybit order from the user's explicit instructions.

The safe default is:

1. Ask for a read-only signal using Bybit market data.
2. Treat the recommendation, confidence, levels, and risk/reward as commentary.
3. If the user asks to analyze, plan, preview, or dry-run, run the order script
   without --execute.
4. If the user asks to open, place, enter, execute, or submit a trade, run the
   order script with --execute and follow the user's requested side, margin,
   leverage, and TP/SL parameters.

Nothing should place an order unless the command includes --execute.


Core Skill
----------

Use the installed skill:

  generating-trading-signals

Signal analysis should follow the core generating-trading-signals skill. This
file only defines the local Bybit data path, symbol format, dry-run flow, and
order safety boundaries for this repo.

Use this workflow for prompts such as:

  get trading signals
  check indicators
  analyze signals

For this repo, use the skill together with these local Bybit scripts:

  scripts/bybit_analyze_symbol.py
  scripts/bybit_demo_open_ethusdt.py

The signal step must stay read-only. The signal does not approve, reject, or
change the user's requested trade direction. The order script should be run
without --execute for plan/dry-run requests and with --execute for explicit
open/place/enter/execute/submit trade requests.

Important Repo Rules
--------------------

Use Bybit symbol format without a slash:

  BTC/USDT -> BTCUSDT
  ETH/USDT -> ETHUSDT
  SOL/USDT -> SOLUSDT

Do not commit .env.

For demo trading, .env should contain:

  BYBIT_BASE_URL=https://api-demo.bybit.com
  BYBIT_API_KEY=your_demo_api_key
  BYBIT_API_SECRET=your_demo_api_secret

or:

  BYBIT_BASE_URL=https://api-demo.bybit.com
  BYBIT_DEMO_API_KEY=your_demo_api_key
  BYBIT_DEMO_API_SECRET=your_demo_api_secret

The read-only analyzer defaults to demo:

  https://api-demo.bybit.com

The order script reads .env and also defaults to demo if BYBIT_BASE_URL is not
set.


Read-Only Signal Command
------------------------

Run the analyzer:

  python3 scripts/bybit_analyze_symbol.py --symbol BTCUSDT

Examples:

  python3 scripts/bybit_analyze_symbol.py --symbol ETHUSDT
  python3 scripts/bybit_analyze_symbol.py --symbol SOLUSDT

Use live public market data explicitly:

  env BYBIT_BASE_URL=https://api.bybit.com python3 scripts/bybit_analyze_symbol.py --symbol BTCUSDT

The analyzer prints:

  symbol
  last_price
  24h_change_pct
  15m_local_high
  15m_local_low
  total_score
  recommendation
  confidence
  15m score, EMA21, EMA55, RSI14
  60m score, EMA21, EMA55, RSI14
  240m score, EMA21, EMA55, RSI14

Basic interpretation for commentary only:

  confidence=strong or moderate -> possible candidate
  confidence=low                -> weaker signal / higher caution
  recommendation=Buy            -> analyzer leans long
  recommendation=Sell           -> analyzer leans short

Prefer trades where 60m and 240m agree with the recommendation, but this is a
commentary note only. Do not use it to block a user-specified order.


Recommended Signal Prompt
-------------------------

Use this prompt when you want a signal:

  Use the generating-trading-signals skill. Analyze BTCUSDT using Bybit API
  market data and the local repo analyzer. Do not place an order. Return:
  recommendation, confidence, 15m/1h/4h scores, RSI/EMA notes, local high/low,
  requested trade direction, invalidation, and risk commentary. Do not approve
  or reject the user's requested trade.

For another symbol, replace BTCUSDT:

  Use the generating-trading-signals skill. Analyze SOLUSDT using Bybit API
  market data and the local repo analyzer. Do not place an order. Return signal
  commentary and explain the risk. Do not approve or reject the user's requested
  trade.


Signal Commentary Checklist
---------------------------

When commenting on a requested order, answer these:

1. Is confidence moderate or strong?
2. Do the 60m and 240m scores agree with the recommendation?
3. Is the symbol liquid enough for the position size?
4. Is the market extended after a large candle?
5. Is there a clear invalidation level?
6. Is the likely reward/risk at least 1.5:1?
7. Is this still demo mode, unless you intentionally switched to live?

These answers are commentary only. Do not convert them into approve/reject,
take/skip, or direction-changing decisions when the user has provided explicit
trade instructions.


Dry-Run Order Plan
------------------

When the user asks for a plan, preview, or dry-run, run a dry-run that follows
the user's parameters. This does not place an order.

When the user asks to open, place, enter, execute, or submit a trade, use the
same parameters and add --execute.

Long example:

  python3 scripts/bybit_demo_open_ethusdt.py --symbol BTCUSDT --side Buy --margin 25 --leverage 5 --tp-sl-mode structure --structure-interval 60

Short example:

  python3 scripts/bybit_demo_open_ethusdt.py --symbol BTCUSDT --side Sell --margin 25 --leverage 5 --tp-sl-mode structure --structure-interval 60

For slower 4h structure:

  python3 scripts/bybit_demo_open_ethusdt.py --symbol BTCUSDT --side Buy --margin 50 --leverage 5 --tp-sl-mode structure --structure-interval 240

The dry-run prints:

  symbol
  side
  margin_usdt
  leverage
  notional_usdt
  entry_reference
  qty
  stop_loss
  take_profit
  target_source
  order_payload


Risk/Reward Check
-----------------

For a Buy:

  risk   = entry_reference - stop_loss
  reward = take_profit - entry_reference

For a Sell:

  risk   = stop_loss - entry_reference
  reward = entry_reference - take_profit

Reward/risk:

  reward / risk

Comment if reward/risk is below 1.5.

Also comment if:

  stop_loss is too close to entry
  take_profit is too close to entry
  stop_loss or take_profit is on the wrong side of entry
  qty is larger than intended
  leverage is higher than intended
  the setup conflicts with the analyzer recommendation

For a Buy, the dry-run must satisfy:

  stop_loss < entry_reference < take_profit

For a Sell, the dry-run must satisfy:

  take_profit < entry_reference < stop_loss


Execution
---------

Execute when the user explicitly asks to open, place, enter, execute, or submit
a trade. The analyzer output is commentary only and must not be treated as
approval or rejection.

Execution adds --execute:

  python3 scripts/bybit_demo_open_ethusdt.py --symbol BTCUSDT --side Buy --margin 25 --leverage 5 --tp-sl-mode structure --structure-interval 60 --execute

If Bybit returns leverage not modified:

  retCode: 110043

rerun with:

  --skip-leverage

Example:

  python3 scripts/bybit_demo_open_ethusdt.py --symbol BTCUSDT --side Buy --margin 25 --leverage 5 --tp-sl-mode structure --structure-interval 60 --skip-leverage --execute


Full Workflow Prompt
--------------------

Use this when you want Codex to do the analysis and prepare a dry-run command,
but not execute:

  Use the generating-trading-signals skill for BTCUSDT. Query Bybit market data
  with the local analyzer, explain the signal as commentary only, and give me
  the exact dry-run command that follows my requested side, margin, leverage,
  and TP/SL parameters. Do not run --execute.

Use this when you want Codex to also run the dry-run:

  Use the generating-trading-signals skill for BTCUSDT. Query Bybit market data
  with the local analyzer. Run the Bybit order script as a dry-run only using
  my requested side, margin, leverage, and TP/SL parameters, calculate
  reward/risk from the printed plan, and provide signal/risk commentary only.
  Do not use --execute.

Use this when you want Codex to open the trade:

  Use the generating-trading-signals skill for BTCUSDT. Query Bybit market data
  with the local analyzer and treat it as commentary only. Open the trade using
  my requested side, margin, leverage, and TP/SL parameters. Include --execute.

Safety Notes
------------

This is not financial advice.

Use demo mode until the process is proven in your own logs.

Never share API secrets in prompts or commit them to git.

The signal can be wrong. Treat it as commentary, not an approval or rejection
gate.

The user's explicit trade instruction controls side, margin, leverage, and TP/SL
parameters. If the user asks to open, place, enter, execute, or submit a trade,
the order command should include --execute.
